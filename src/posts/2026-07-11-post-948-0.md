### Memahami Konsep Circuit Breaker Digital
Macam yang saya dah cerita tadi, circuit breaker digital ni memang sama je konsepnya dengan fius kat rumah kita. Bila kita 'pasang' fius kat litar, ia dalam keadaan 'tertutup' (closed), maksudnya, elektrik boleh mengalir. Kalau ada `overload` atau litar pintas, fius tu akan 'buka' (open), putuskan litar nak lindungi peralatan kita. Dalam dunia coding, konsep ni diterjemahkan kepada 3 'status' utama:

1.  **Closed (Tertutup):** Ini adalah keadaan normal. Semua permintaan kita boleh lalu ke API luar. Tapi kalau ada kegagalan, 'kaunter' kegagalan akan naik satu.
2.  **Open (Terbuka):** Bila 'kaunter' kegagalan tu dah cecah satu `threshold` yang kita tetapkan, circuit breaker akan terus 'terbuka'. Maknanya, semua permintaan seterusnya ke API tu akan dihalang serta-merta, dan kita akan dapat balik ralat tanpa perlu buat panggilan sebenar. Ini bagi peluang kat API tu untuk bertenang dan pulih. Kira macam bagi dia rehat sekejap lah.
3.  **Half-Open (Separuh Terbuka):** Lepas circuit breaker dah 'terbuka' untuk satu tempoh masa (biasanya beberapa saat atau minit), ia akan tukar ke status 'separuh terbuka'. Dalam keadaan ni, cuma satu atau beberapa permintaan je yang dibenarkan untuk lalu. Kalau permintaan ni berjaya, hooray! Circuit breaker akan balik ke status 'tertutup' semula. Tapi kalau gagal, aduh, dia akan balik ke status 'terbuka' lagi untuk tempoh yang lebih lama.

Memang penting gila untuk faham kitaran ni kalau nak buat circuit breaker yang berkesan.

### Langkah 1: Mengenal pasti Keadaan Kegagalan dan `Threshold`
Sebelum kita boleh pasang circuit breaker ni, kita kena lah tahu bila dia patut 'trip', kan? Untuk DeepSeek API, jenis-jenis kegagalan yang boleh buat dia 'trip' termasuklah:
*   **Ralat HTTP:** Macam kod status `5xx` (Server Error) atau `429` (Too Many Requests) bila server DeepSeek tu ada masalah atau kita hantar banyak sangat permintaan.
*   **`Timeout`:** Bila panggilan API kita tu ambil masa terlalu lama nak jawab. Kita tak boleh tunggu sampai esok kan?
*   **Ralat Rangkaian:** Isu dengan sambungan internet kita atau masalah DNS. Ini selalu jadi punca juga.
*   **`Parsing Error`:** Kalau respons dari API tu tak boleh diproses dengan betul. Kadang-kadang format data lari.

Lepas tu, kita kena tetapkan juga `threshold` (ambang) dan `timeout` untuk 'berehat'. Contohnya, kita boleh set macam ni:
*   **`Failure threshold`:** Katalah, 5 kali gagal dalam masa 60 saat.
*   **`Timeout` Terbuka:** 30 saat (tempoh circuit breaker tu akan 'terbuka' dan tak benarkan sebarang panggilan).

### Langkah 2: Membina Logik Circuit Breaker Asas
Untuk membina logik circuit breaker ni, anda boleh guna bahasa pengaturcaraan apa saja yang anda selesa. Kebanyakan bahasa memang dah ada `library` atau `framework` yang sedia ada, contohnya `Hystrix` untuk Java, `Polly` untuk .NET, atau yang custom untuk Python. Kalau takde pun, jangan risau, kita boleh tulis sendiri! Bukan lah susah sangat.

Ini adalah kerangka idea asas macam mana nak buat dalam Python:

```
# Inisialisasi: status=CLOSED, failure_count=0, last_failure_time=None, open_timeout=30s, failure_threshold=5

# Apabila panggilan API dibuat:
#   Jika status=OPEN dan masa kini < last_failure_time + open_timeout:
#     Pulangkan ralat 'Circuit is Open'.
#   Jika status=HALF_OPEN:
#     Cuba satu panggilan. Jika berjaya, status=CLOSED, failure_count=0. Jika gagal, status=OPEN, last_failure_time=masa_kini.
#   Jika status=CLOSED atau (status=OPEN dan masa kini >= last_failure_time + open_timeout):
#     Buat panggilan API ke DeepSeek.
#     Jika berjaya: failure_count=0.
#     Jika gagal: failure_count++. last_failure_time=masa_kini.
#     Jika failure_count >= failure_threshold:
#       status=OPEN, last_failure_time=masa_kini.
```

Penting juga untuk pastikan kod anda pandai handle `rate limits` DeepSeek API. Gabungan strategi `retry` dan circuit breaker ni memang padu gila! Kalau nak tahu lebih dalam pasal strategi `retry`, jom tengok panduan kami kat sini: [Belajar Strategi Retry untuk DeepSeek API](/posts/deepseek-api-retry-guide/).

### Langkah 3: Menambah Sistem Amaran dan `Logging`
Circuit breaker ni memang akan berfungsi secara automatik, tapi kita kena tahu juga bila dia 'trip' atau bila dia dah pulih, kan? Sebab tu lah sistem amaran ni penting sangat! Kita boleh set macam-macam:

*   **Log:** Rekod semua perubahan status circuit breaker (`OPEN`, `HALF_OPEN`, `CLOSED`) dalam log sistem kita. Ini memang sangat membantu bila nak cari punca masalah nanti.
*   **E-mel/SMS:** Hantar amaran terus ke pasukan teknikal kita bila circuit breaker 'terbuka'. Cepat dapat tahu, cepat bertindak.
*   **Integrasi Chat:** Hantar notifikasi ke platform macam Telegram atau Slack. Ini lagi senang sebab pasukan boleh terus berbincang dan bertindak balas sepantas kilat.

Bagi saya, sistem amaran ni memang sangat membantu untuk kenal pasti masalah dengan API luar, atau kadang-kadang isu dalaman yang jadi punca kegagalan API. ### Langkah 4: Menguji dan Menyelaraskan Circuit Breaker Anda
Pasang circuit breaker tanpa uji, samalah macam pasang fius tapi tak tahu berfungsi ke tak. Bahaya! Jadi, anda kena lah buat simulasi kegagalan untuk tengok macam mana circuit breaker kita ni bertindak balas:

*   **Simulasi `Timeout`:** Cuba set `timeout` panggilan API kita ke nilai yang super rendah, lepas tu tengok circuit breaker tu 'trip' ke tak.
*   **Simulasi Kegagalan API:** Buat satu kod yang sengaja return error HTTP `500` atau `429` dari `mock server` atau `endpoint` ujian. Perhatikan `failure_count` dan perubahan status. Ini memang cara paling efektif.
*   **Ujian `Half-Open`:** Bila circuit breaker dah 'terbuka', tunggu sampai `open_timeout` tamat, lepas tu hantar satu dua permintaan. Tengok dia pulih ke `CLOSED` balik, atau dia jadi `OPEN` semula. Penting ni!

Penyelarasan ni maksudnya kita kena ubah-suai `failure_threshold` dan `open_timeout` berdasarkan cara aplikasi kita guna API dan tahap risiko yang kita sanggup terima. Ingat, setiap aplikasi unik, jadi takde lah 'satu saiz sesuai untuk semua' orang. Kita mungkin perlu pantau log prestasi dan buat `adjustment` dari masa ke masa. Dengan cara ni, kita juga boleh urus [batasan guna DeepSeek API](/posts/deepseek-api-key-and-limits/) dengan lebih baik.

## Soalan Lazim (FAQ)

**S1: Apa itu circuit breaker dalam konteks API DeepSeek?**
J: Circuit breaker ialah corak reka bentuk perisian yang digunakan untuk mengelakkan aplikasi daripada berulang kali cuba melaksanakan operasi yang berkemungkinan gagal. Untuk DeepSeek API, ia menghentikan sementara permintaan kepada API DeepSeek apabila ia mengesan kegagalan berulang, bagi membolehkan API pulih dan mencegah sistem kita daripada terus membuat panggilan yang sia-sia.

**S2: Kenapa circuit breaker penting untuk aplikasi fintech saya di Malaysia?**
J: Dalam sektor fintech, ketersediaan dan kestabilan adalah sangat penting. Circuit breaker melindungi aplikasi anda daripada kegagalan API luaran yang boleh menyebabkan `downtime`, `timeout`, dan pengalaman pengguna yang tak best. Ini memastikan perkhidmatan kewangan anda kekal stabil walaupun DeepSeek API atau API lain mengalami masalah sementara.

**S3: Adakah DeepSeek API mempunyai circuit breaker terbina?**
J: Setakat versi `v1.2.3` SDK DeepSeek API, circuit breaker tidak terbina secara langsung. Pembangun perlu melaksanakannya sendiri di peringkat aplikasi atau menggunakan perpustakaan pihak ketiga untuk fungsi ini. Anda boleh mendapatkan maklumat lanjut mengenai API di laman web rasmi mereka di [platform.deepseek.com](https://platform.deepseek.com).

**S4: Bagaimana PKS di Malaysia boleh menggunakan circuit breaker ini untuk kelebihan mereka?**
J: PKS selalunya ada sumber IT yang terhad. Dengan melaksanakan circuit breaker, PKS dapat kurangkan kerja `monitoring` dan intervensi manual apabila API pihak ketiga ada masalah. Ia membantu mengekalkan operasi perniagaan yang lancar, melindungi reputasi, dan mengurangkan kos penyelenggaraan dengan mengautomasikan pengurusan kegagalan API.

**S5: Apa beza circuit breaker dengan strategi `retry` (cuba semula)?**
J: Strategi `retry` bertujuan untuk cuba semula panggilan API yang gagal dengan harapan ia akan berjaya pada percubaan seterusnya, selalunya dengan `delay` antara percubaan. Circuit breaker pula menghentikan semua panggilan API untuk tempoh masa tertentu selepas sejumlah kegagalan dikesan. `Retry` sesuai untuk kegagalan sementara yang cepat pulih, manakala circuit breaker sesuai untuk kegagalan yang berterusan dan memerlukan masa untuk pulih. Kedua-duanya adalah pelengkap dan boleh digunakan bersama untuk daya tahan sistem yang lebih baik.
